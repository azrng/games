# AGENTS.md

## 目标
- `AGENTS.md`：只保留全局协作规范、阶段门控、任务机制、交付要求与子规则导航
- `frontend-AGENTS.md`：页面、布局、组件、样式、交互与前端测试
- `doc-AGENTS.md`：设计文档、设计系统
- `CLAUDE.md` / `GEMINI.md`：入口提示文件，只提醒先读 `AGENTS.md`
- `design-system.yaml`：设计 token 与 UI 规范
- `TASK.md`：唯一活动任务记录文件

---

## 快速命中卡片
- 只改规则、文档、设计说明：读 `doc-AGENTS.md`
- 改页面、布局、样式、动效、交互：读 `frontend-AGENTS.md`
- 静态页联调、构建打包或交付验证：先读当前文件，再按需回看 `frontend-AGENTS.md`
- 若只是很小的文案、样式或规则修订，命中后只读对应子规则的相关段落，不必通读全文

## 最小充分上下文
### 判断标准定义
- **简单问题**：可在5分钟内回答，无需代码修改的问题（如：解释概念、查询配置、查看现有实现）
- **单点低风险改动**：只涉及单一文件，修改内容少于20行，不影响 API 契约、数据库结构、权限模型或发布链路
- **微小调整**：文案修改、规则措辞微调、注释修正等不影响执行逻辑的改动，修改内容少于10行
- **高风险内容**：涉及数据库迁移、缓存策略变更、鉴权模型变更、发布链路调整、环境变量变更或第三方集成
- **无法继续（BLOCKED）**：遇到明确的技术障碍、依赖缺失、需要用户决策或外部资源
- **开发完成（REVIEW）**：代码或规则实现完成，已通过最小自查，等待用户确认或代码审查

- 简单问题、纯问答、方案判断：只加载回答当前问题所需的最小规则与文件，不默认预读全部子规则
- 单点低风险改动：先读当前文件，再读命中的一个子规则和目标文件；若信息已足够，不继续扩展上下文
- 只有在以下情况才继续补读更多规则或实现文件：
  - 需要实际落文件
  - 任务跨设计、页面实现与交付验证
  - 涉及构建、部署、资源引用或高风险页面重构
  - 当前信息不足以判断影响范围
- 若任务从简单问题升级为真实修改，应按命中场景逐步补充上下文，而不是一开始加载全部信息

## 根文件原则
- 根 `AGENTS.md` 只放所有任务都需要知道的规则，不重复展开前端与文档细节
- 只有任务命中对应场景时，才继续阅读对应子规则文件
- 用户指令始终高于本文件
- 先理解业务目标与现有实现，再修改代码或规则
- 优先复用现有结构，避免无关重构和过度设计
- 优先做可验证的改动；无法验证时必须说明原因、影响和建议下一步

4. 再阅读相关页面与资源文件，确认复用点后再动手。

## 开发阶段

### 三个阶段

```text
阶段 0          阶段 1                阶段 2
设计文档   →   页面与交互实现     →   联调与交付
（用户确认）    （页面实现角色）        （协作联调角色）
```

- 新功能需经过完整阶段流程，以下场景可简化或跳过阶段 0：
  - Bug 修复
  - 已有页面的样式或文案微调
  - 资源引用、构建配置或部署说明调整
- 除上述场景外，各阶段必须满足入场条件后方可进入下一阶段
- 阶段括号仅表示推荐职责角色，不绑定具体 AI 工具；单 Agent 场景下可在满足门控条件后连续执行各阶段

### 阶段职责
- 阶段 0：需求整理、设计文档与页面说明，细则见 `doc-AGENTS.md`
- 阶段 1：页面骨架、交互、视觉落地与前端验证，细则见 `frontend-AGENTS.md`
- 阶段 2：联调、构建、部署说明与交付验证，结合当前文件与命中的子规则执行

## 全局工作规则
- 先理解，再修改；不要基于猜测改代码
- 优先通过页面状态、资源引用、构建结果与 smoke test 等证据定位问题
- 先复用，再新增；不新增平行样式体系，不绕过现有结构
- 页面交付必须真实可演示，不能只停留在静态截图层面
- 输出保持简洁，但分析、实现、验证要完整
- 已确认无变更风险的文件无需重复阅读；出现上下文缺口时必须补读
- 删除文件、删除目录、批量重构、资源清理等高风险操作，必须先获得明确确认

### 理解检查清单
在开始真实修改前，至少确认以下信息；不相关项可跳过，但不能跳过命中项：
- 已阅读用户目标和当前任务状态，明确本次是问答、评审、文档还是页面改动
- 已阅读命中的子规则和目标文件
- 涉及页面时，已查看相关页面结构、布局、样式资源、设计 token、资源引用和现有交互模式
- 已识别可复用实现、影响范围、验证方式和仍需确认的问题

### 查证优先级
不确定 API、依赖版本、文件是否存在、业务规则、配置项或命令结果时，按以下顺序查证：
1. 先检索项目代码、配置和测试。
2. 再读取相关实现文件、文档、任务记录和开发记录。
3. 再查看项目锁定的依赖版本、包配置或官方文档。
4. 仍无法确认时，列为待确认项并询问用户；不得把假设写成事实。

### 纠错与回退机制
- 阶段 0 输出的页面说明、设计规范或字段定义在阶段 1 发现缺口时，应先回到设计变更流程，修改设计说明 / 页面规范后同步页面实现和相关文档，再继续实现；不得绕过设计直接补临时样式
- 发现前一阶段产物与需求、设计文档或真实实现冲突时，先说明冲突、影响范围和建议回退点，再按最小影响范围修正
- 自动提交后发现问题时，优先追加修复提交；只有用户明确要求且确认影响范围后，才执行回滚类 Git 操作
- 涉及发布产物、环境清理的回退，必须先确认备份、回滚窗口和可验证恢复方式
- 纠错过程需要更新 `TASK.md` 的状态、阻塞项或交接说明

### 技术债务与重构判断
- **沿用现有实现**：代码可正常运行、无明显性能问题、维护成本可接受、团队熟悉度高
- **必须重构**：存在安全漏洞、严重影响性能、阻碍新功能开发、维护成本过高
- **可选重构**：代码风格不一致、存在更好的替代方案，但不紧急
- **禁止重构**：仅为个人偏好、追赶技术潮流、非关键路径的过度优化

重构决策流程：
1. 先评估现有代码的技术债务等级（高 / 中 / 低）。
2. 判断重构的紧急程度和业务价值。
3. 评估重构风险和工作量。
4. 高风险或大规模重构需先与用户或团队确认。

## 模型行为约束与主动建议
- 不确定 API、依赖版本、文件是否存在、业务规则、配置项或命令结果时，必须按"查证优先级"处理；禁止编造接口、配置、测试结果、文件路径和业务结论
- 新增页面或较大改动前，必须先检索相似页面、布局、组件、样式资源和交互模式，确认复用点后再新增
- 修改应保持最小扰动，不做无关格式化、无关重构、无关依赖升级、无关技术方案替换或大面积重排
- 修 Bug 必须定位根因并验证原始复现场景；禁止通过吞异常、随意空判断、绕过校验或包裹 try-catch 掩盖问题
- 涉及页面结构、资源引用、样式体系或交互逻辑时，必须先判断架构边界，避免职责混杂
- 发现更简单实现、已有复用点、需求缺口、验证缺口或潜在风险时，必须主动提醒
- 主动建议是风险提示和最小方案，不是自动扩大任务范围；未经用户确认，不得把建议项变成本次额外实现
- 主动建议需区分"必须处理""建议处理""暂不处理"：必须处理的问题说明原因、影响和方案；建议处理的问题给出最小改法；暂不处理的问题记录风险即可

## 编码与文档规则
- 所有源码、配置、文档统一使用 `UTF-8`
- 读取或修改含中文文件时，如出现乱码，先判断是显示问题还是文件损坏，未确认前禁止覆盖
- Windows / PowerShell 下读取中文文件必须显式使用 `UTF-8`
- Windows / PowerShell 下处理中文 Git 提交信息时，优先使用 `git commit -m` / 多个 `-m` 参数，或使用 `UTF-8` 编码文件配合 `git commit -F <file>`
- 禁止通过标准输入管道直接传递中文提交信息，如 `git commit -F -`、here-string 管道等，避免编码链路把中文写成 `?`
- 禁止使用可能隐式改变编码的方式直接改写源码文件，如 shell 重定向、`Out-File`、`Set-Content`
- 修改含中文内容后，必须重新读取确认关键中文显示正常

## 任务管理
- `TASK.md` 是唯一活动任务记录文件，只维护最小任务记录与当前仍需关注的任务
- 开始工作前先检查是否已有任务；已有则更新原任务，没有则新增
- 只读评审、纯问答、现状分析且不落文件时，可不更新 `TASK.md` 和提交记录；若分析后转为规则或代码修改，则按正常任务流程补齐
- 任务状态仅使用：`TODO`、`DOING`、`BLOCKED`、`REVIEW`、`DONE`

### 任务状态触发条件
- `TODO`：任务目标已明确，但尚未开始分析、修改或验证
- `DOING`：已开始分析、修改或验证，任务正在持续推进
- `BLOCKED`：遇到明确技术障碍、依赖缺失、需要用户决策或外部资源，当前无法继续推进
- `REVIEW`：实现或规则修订已完成，且已完成最小自查，等待用户确认或代码审查
- `DONE`：验证完成，结论明确，且当前无需继续跟进

- 开始执行时更新为 `DOING`
- 无法继续时更新为 `BLOCKED` 并写明原因
- 开发完成待确认时更新为 `REVIEW`
- 验证完成并确认交付后更新为 `DONE`
- 每条任务至少包含：任务 ID、任务名称、任务状态、最近更新时间
- 跨 Agent 或跨会话任务应补充：已完成步骤、下一步、阻塞项或交接说明，避免后续 Agent 重新推断上下文

### 任务清理规则
- `TASK.md` 默认只保留：`TODO`、`DOING`、`BLOCKED`、`REVIEW`，以及最近 5 条 `DONE` 任务
- `BLOCKED`、`REVIEW` 任务不得删除，避免仍需处理的事项从主表消失
- `DONE` 任务超过 5 条时，自动删除最早的 `DONE` 任务
- 不再使用 `doc/task-archive/` 归档

### 文档同步要求
- 设计说明、资源引用、构建发布流程发生变化时，必须同步更新相关文档
- 设计文档细则见 `doc-AGENTS.md`

## Git 与交付

### Commit 规范
- 提交信息优先采用 Conventional Commits
- Commit 标题格式：`<type>(<scope>): <中文简述>`
- Windows / PowerShell 下提交前，优先确保当前终端与 Git 日志输出使用 `UTF-8`；若刚执行过中文提交，需回读最近一次提交信息确认标题与正文未乱码
- Commit 正文默认使用中文；非微小调整应补充以下字段：
  - `任务`
  - `阶段`
  - `原因`
  - `变更`
  - `风险`
  - `关联`
  - `AI`
- Commit 正文中的 `风险` 仅记录真实的代码、业务、数据、兼容性、部署或发布风险，不写过程性说明
- 未执行人工测试、未补 smoke test、未覆盖某项校验、仅做等价验证等过程性说明，统一写在最终输出，不写入 Commit 正文
- 提交前确认 `TASK.md` 已同步；若未同步，需说明原因
- push、PR / MR、交付说明应优先复用最近一次 Commit 正文中的结构化字段，并使用中文摘要


### 提交触发规则
- 当用户请求”开始开发”而非”只出方案”时，若本次代码、配置或规则改动已完成，且已完成本次改动所需验证，并已将 `TASK.md` 中对应任务状态更新为 `DONE`，则默认自动执行一次提交
- 自动提交前必须确认提交信息符合本节 `Commit 规范`
- 若验证尚未完成、任务状态尚未更新、仍存在明确阻塞，或用户明确要求暂不提交，则不得自动提交；需先说明原因与当前状态

### 提交前最小回归
- 文档或规则文件改动：重新读取改动文件，确认 `UTF-8` 与关键中文正常；无需强制执行 `lint` / `build`
- 页面、样式、交互改动：至少执行受影响页面或组件的 smoke test / 交互验证
- 构建配置、资源引用或发布链路改动：在前端回归基础上，再确认构建、资源加载与交付文档已同步
- 若仓库当前缺少对应脚本，可使用“等价验证”替代，但必须在最终说明与 `doc/devlog/` 中写清执行内容、未覆盖校验和后续建议

### 交付前自查
- 根据改动类型完成”提交前最小回归”
- 涉及构建链路时执行 `build`
- 涉及页面访问链路时补关键路径 smoke test
- 设计说明、文档、`TASK.md`、忽略文件按需同步

### 交付收口
- 当用户请求”开始开发”而非”只出方案”时，实现完成后默认继续做交付收口
- 交付收口至少按实际情况检查：
  1. 测试 / 等价验证
  2. 构建
  3. 发布产物与启动验证
  4. 应用级 smoke test
  5. 文档、`TASK.md`、忽略文件更新
  6. 满足”提交触发规则”时完成自动提交
- 若某项验证未执行，必须明确说明未执行项、原因、影响范围和建议下一步

### 最终输出要求
- 最终输出优先使用中文，并至少说明：
  - 本次改了什么
  - 核心实现方式
  - 修改了哪些文件
  - 已执行和未执行的校验
  - 当前任务状态
  - 风险、阻塞或假设

## 重复错误回顾机制
- 同一类问题因规则缺口在同一任务中连续两次阻塞交付，或跨任务重复出现时，必须做一次简短回顾
- 回顾至少说明：错误现象、根因、规则缺口、后续如何避免
- 回顾结论必须落实到当前架构目录的 `AGENTS.md` 或对应子规则文件中
