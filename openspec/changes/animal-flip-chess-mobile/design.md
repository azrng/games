## Context

动物翻翻棋游戏计划实现移动端适配。当前游戏设计面向桌面端，使用鼠标点击交互。移动端用户占比高，需要确保触控操作的流畅性和准确性。

项目使用纯 HTML/CSS/JS 技术栈，无框架依赖。现有游戏（如五子棋、找猫咪）已有移动端适配经验可参考。

## Goals / Non-Goals

**Goals:**
- 实现移动端响应式布局，适配 320px-428px 宽度范围
- 优化触控交互，支持点击翻牌和拖拽移动
- 确保横竖屏切换时布局正常
- 保持与桌面端一致的游戏体验

**Non-Goals:**
- 不实现原生 App 功能（PWA、离线等）
- 不适配平板设备（仅手机优先）
- 不修改游戏核心规则逻辑

## Decisions

### 1. 响应式方案：CSS 媒体查询 + Flexbox

**选择**: 使用 CSS 媒体查询配合 Flexbox 布局
**理由**: 项目已使用纯 CSS，无需引入框架。Flexbox 能很好地处理棋盘和 UI 元素的自适应排列
**替代方案**: CSS Grid（兼容性稍差）、固定尺寸+缩放（体验不佳）

### 2. 触控交互：原生 Touch 事件

**选择**: 使用 touchstart/touchmove/touchend 原生事件
**理由**: 无需引入第三方库，性能好，与现有鼠标事件处理逻辑可复用
**替代方案**: Hammer.js（过重）、Pointer Events（兼容性考虑）

### 3. 棋盘尺寸：视口单位 + 最大限制

**选择**: 使用 vw/vh 单位设置棋盘尺寸，配合 max-width 限制最大尺寸
**理由**: 自动适配不同屏幕，同时保证大屏设备上的合理显示
**替代方案**: 固定像素值（需多套断点）、JavaScript 动态计算（复杂）

### 4. 横竖屏适配：orientation 媒体查询

**选择**: 使用 orientation 媒体查询检测屏幕方向，调整布局
**理由**: 简单可靠，CSS 原生支持
**替代方案**: JavaScript 监听 resize 事件（性能开销）

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 触控精度不足导致误操作 | 增大可点击区域，添加触控反馈动画 |
| 小屏幕设备显示空间不足 | 优化 UI 紧凑度，必要时隐藏非核心信息 |
| 横屏模式体验不佳 | 优先保证竖屏体验，横屏作为可选 |
| 不同浏览器触控行为差异 | 充分测试主流浏览器（Chrome、Safari、微信内置浏览器） |
